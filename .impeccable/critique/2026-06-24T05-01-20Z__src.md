## Anti-Patterns Verdict

**LLM 评审**：整体不"AI 味"，一个熟悉 Linear/Stripe/Raycast 的用户会大体信任，但在三处会停顿：
- Hero-metric 模板（命中）：DashboardPage.tsx 的 StatTile（行 12-22）是"大数字+小标签+图标"的 SaaS 套路，4 个 StatTile + 3 个同结构 Panel 是产品 UI 里最模板化的部分。
- 完全相同卡片网格（命中）：Dashboard 第二行 3 个 Panel（今日时长/本月时长/本周费用）结构完全一致，仅标签数字不同。
- 小号大写字距 eyebrow（命中）：CreateTaskDialog.tsx 的 FormSection legend 用 text-xs font-medium uppercase tracking-wide text-app-muted，每个分组上方都是这套大写字距小字。表单 legend 不需要大写字距。
- 1px border + 宽 box-shadow 配对（命中）：styles.css 的 .app-modal-panel（行 154-161）同时有 border: 1px solid 和 box-shadow: 0 24px 64px，是 codex"幽灵卡片"tell。模态可保留宽阴影，但应去掉 1px 边框。
- 背景色 oklch(0.982 0.006 95)：L 0.982 略高于奶油带，C 0.006 几乎中性，读作"近乎纯白+一丝暖意"，DESIGN.md 明确要求 slightly warm neutrals，不算 tell。
- 侧边条纹、渐变文字、玻璃拟态、编号标记、手绘 SVG、重复线性渐变条纹：均无。

**确定性扫描**：检测器扫描整个 src 目录，返回 1 条发现：
- overused-font × 1 — src/styles.css:28，font-family: Inter。Inter 是规则点名的过度使用字体之一。真阳性（已核对源码）。

**LLM 与检测器的分歧**：检测器标记 Inter 为过度使用字体，但 LLM 评审认为 Inter 单字族符合产品 UI 规范（product register 明确允许 system fonts 和 Inter/SF Pro 栈）。这是合理的分歧——Inter 在产品 UI 里是"earned familiarity"，但在品牌向页面里会是 tell。鉴于 EasyConsole 是 product register，建议保留 Inter 但知悉此标记；若想提升辨识度可考虑切换到同样中性但更少见的字族。

**可视化叠加**：本会话无浏览器自动化工具，未启动 dev server、未注入脚本。回退信号：仅 CLI 检测器。无用户可见 overlay。

## Overall Impression

这是一个认真做过的操作工具，状态处理和桌面/移动双布局是真正的强项。但本轮深入审查后发现一致性是当前最大短板：同一类操作（三个任务表单）用三套校验模式，同一概念（"更多"）用两套 ARIA 语义，同一术语（Commit）在中英文间硬切换，同一确认交互（Settings vs 全应用）用 window.confirm vs ConfirmDialog。这些不是装饰问题，是产品专业度的裂缝——用户会在每次切换页面时感到"规则变了"。

第二大问题是错误恢复停留在"报错"而非"帮修"：原始异常字符串当错误文案、表单错误远离字段、Commit 长耗时操作无进度。操作工具的用户最焦虑的时刻就是出错时，当前界面在这些时刻没有给足安心感。

最大单一机会：把三个任务表单的校验抽成共享 hook/组件，一次性消灭一致性缺陷 + 重复代码 + 错误恢复不贴字段三个问题。

## What's Working

1. 状态覆盖与操作反馈是真正的强项。每个异步界面都有 loading（skeleton/spinner）、empty、error（带重试）、success（toast）四态；auto-refresh 在对话框打开时自动暂停；上传/下载队列有进度/重试/取消/清理；断网横幅、列设置持久化、pin 持久化。DataState.tsx + Toast.tsx + useConfirmAction + 下载队列构成完整操作反馈骨架，用户几乎不会被"卡住不知道发生了什么"。

2. 桌面/移动双布局是认真做的，不是缩水。六张表页都有平行移动卡片布局，动作一致；移动底部导航 + More sheet + safe-area inset + 44px 触控目标 + xterm 移动键栏（Esc/Tab/方向键/Ctrl + Ctrl+C/Z/D/L）显示了对移动终端场景的真实思考。AppSshTerminalDialog 是全应用最打磨的瞬间。

3. CommandPalette 与 MoreActionsMenu 的键盘/ARIA 工艺高于平均。CommandPalette 有完整 combobox 语义（aria-activedescendant、ArrowUp/Down/Home/End/Enter、isComposing 防中文输入误触）；MoreActionsMenu 有 role=menu/menuitem、箭头键导航、Home/End、Esc 还原焦点、滚动关闭、定位翻转。这两处是产品 UI 里少见的"做对了"的实现。

## Priority Issues

### [P1] "Commit" 三处硬编码英文，破坏中文优先与双语体系
- 位置：TasksPage.tsx 行 437（title）、445（菜单项标签）、814（confirmCommitTask 的 confirmLabel）
- Why it matters：整条菜单和确认弹窗都是中文，唯独 Commit 是英文。用户在确认弹窗看到"将把 xxx 的当前运行环境提交为镜像"中文描述，按钮却写 "Commit"，割裂感强。且与已走 i18n 的 toast text("Commit 已提交", "Commit submitted") 不一致。v2 已标为 P3 但未修。
- Fix：三处统一改为 text("提交镜像", "Commit")。
- Suggested command: clarify

### [P1] 三个任务表单的校验模式不统一，两处缺 inline 校验
- 位置：CreateTaskDialog.tsx（有 inline：fieldErrors+touchedFields+FieldError）vs ScheduledTasksPage.tsx（行 150-203，仅顶部 formError）vs TaskTemplatesPage.tsx 的 TemplateDialog（行 120-175，仅底部 formError）
- Why it matters：用户在定时任务/模板表单填错时，只看到表单底部一条红字，必须自己回找哪个字段错了。CreateTaskDialog 已证明更好的模式存在。这是"同一类操作三种行为"的一致性缺陷，也是错误预防缺口。
- Fix：把 CreateTaskDialog 的 fieldErrors+touchedFields+FieldError 抽成共享 hook/组件，应用到另两个表单。
- Suggested command: harden

### [P2] 移动 More 菜单与"视图"下拉的 ARIA 语义和键盘导航缺失
- 位置：AppShell.tsx 移动 More sheet（行 418-445）是 nav + NavLink，无 role=menu/menuitem、无箭头键、无焦点陷阱；TasksPage.tsx 行 1080"视图"按钮 aria-haspopup="true" 但弹出的是设置面板（checkbox+select+按钮），语义错配
- Why it matters：键盘/屏幕阅读器用户在移动端无法高效操作 More sheet；同应用内两套"更多"交互不一致。v2 已标为 P1 但未修。
- Fix：More sheet 升级为 role=menu+menuitem+箭头键+焦点陷阱（对齐 MoreActionsMenu），或改 aria-haspopup="dialog"+焦点陷阱；"视图"下拉改 aria-haspopup="dialog"。
- Suggested command: harden

### [P2] 表格数据单元格滥用 text-app-muted，导致扫读灰平
- 位置：TasksPage.tsx（resource/node/owner/group/endpoint/duration/created/release）、StoragePage.tsx（type/size/modified）、ImagesPage.tsx（source/tag/description/updated）、RunLogsPage.tsx（time/channel/target/duration）、TaskTemplatesPage.tsx（image/resource/paths/release）的数据值都用 text-app-muted
- Why it matters：用户来读的就是这些数据，但它们和表头/次要说明同灰度。表格没有"数据浮起"的层级，扫读时所有信息等权重。DESIGN.md 说"scan-friendly contrast"，实际效果相反。
- Fix：数据值改 text-app-text，仅时间戳/ID/描述等真正次要信息保留 text-app-muted；表头保持 muted。
- Suggested command: colorize

### [P2] Commit 长耗时操作无进度反馈，是情感谷底
- 位置：TasksPage.tsx confirmCommitTask 流程
- Why it matters：Commit 是全应用唯一不可逆、长耗时、无进度的操作。确认弹窗说"此操作可能需要一段时间"，确认后只有一个 toast"Commit 已提交"，之后沉默。用户不知道何时完成、是否成功。对比下载队列有进度条，Commit 是黑盒。
- Fix：像下载队列一样开一个持久进度面板/任务卡片，轮询状态直到完成或失败。
- Suggested command: harden

### [P3] EmptyState 无视觉差异，所有空状态长一样
- 位置：DataState.tsx 的 EmptyState（行 42-49）只有居中灰字+可选动作，无图标无情境区分
- Why it matters：空状态是教学时机。用户无法一眼区分"从未有数据"vs"搜索无结果"vs"出错"。v2 已标为 P3 但未修。
- Fix：按情境加 lucide 图标（Inbox=无数据、SearchX=无结果、FolderOpen=空目录），单图标即可，不要插图。
- Suggested command: onboard

## Persona Red Flags

### Alex（重度用户，追求效率）
主路径：打开 Tasks → 找运行中实例 → 开终端 → 监控。
- 红旗 1：全应用只有 Ctrl+K 一个快捷键。无 n 新建、/ 搜索、r 刷新、t 在选中行开终端。Alex 必须为每个动作动鼠标。
- 红旗 2：每行 6 个 icon 按钮无行级快捷键。Tab 遍历 50 行表格去按第 30 行的终端按钮不现实。
- 红旗 3：复选框只能鼠标点，无 Shift+Up/Down 多选。
- 红旗 4：命令面板只导航页面和搜任务，不能执行动作（释放/删除/Commit/开终端）。Alex 无法从面板驱动整个应用。
- 红旗 5：auto-refresh 最快 5 秒，监控部署时想要 2-3 秒做不到。

### Sam（依赖键盘与屏幕阅读器）
主路径：Tab 进入 Tasks → 选中任务 → 查看日志。
- 红旗 1：移动 More sheet 打开后焦点留在"更多"按钮，Sam 必须 Tab 进 sheet 且不知焦点在哪。无焦点陷阱、无箭头键（见 P2）。
- 红旗 2：TasksPage 行内 6 个 icon 按钮，长表格 Tab 遍历会有数百个 tab stop。无"跳到操作"或行级动作菜单默认。
- 红旗 3：ScheduledTasksPage/TaskTemplatesPage 表单错误在顶部/底部，不贴字段。Sam 用读屏要来回找哪个字段错。
- 红旗 4：多数错误是原始异常字符串（HTTP 500 级），读屏读出来对 Sam 无意义。
- 红旗 5：表格表头不可排序，Sam 无法用键盘重排数据。

### Riley（压力测试，推边缘）
主路径：用极端输入建任务 → 测上传失败 → 中途刷新。
- 红旗 1：CreateTaskDialog 的 batchCount 虽有 min=1/max=50，但 type="number" 浏览器未必拦。输 0 或 100 只在提交时 formError 报错，无 inline 拦截。
- 红旗 2：StoragePage 文件夹上传用 webkitdirectory，丢一个 10k 文件夹会无预览无限制地撑爆上传队列，UI 卡死。
- 红旗 3：ScheduledTasksPage 的 scheduleTime 可设过去时间，无警告，任务立即变"到期"。
- 红旗 4：SettingsPage 保存畸形 URL 后 queryClient.clear() 静默清缓存，全页故障直到手动改回。
- 红旗 5：RemoteStoragePicker 选了文件后点"上级"会无提示清空选择。

## Questions to Consider

1. 每行 6 个始终可见的动作按钮（监控/日志/终端/复制/释放删除/更多）。如果只留"释放/删除"一个主操作始终可见，其余 5 个收进 More 菜单，表格会不会扫得更快、行高更紧？
2. Dashboard 是 7 个 stat tile + 最近任务表。但操作者真正想看的是"现在什么在跑、什么坏了、什么快释放了"。如果 Dashboard 改成按状态分组的任务列表（运行中/排队/失败/即将释放）开头，会不会比指标卡更有用？
3. Commit 是全应用唯一不可逆、长耗时、无进度的操作。如果 Commit 像下载队列一样开一个持久进度面板，用户能否不再焦虑"它到底完没完"？
4. 三个表单（Create/Scheduled/Template）收集几乎相同的字段，却用三套校验模式。如果抽一个共享 TaskFields 组件 + 一套校验模型，是不是同时消灭了一致性缺陷和重复代码？
5. 移动 More sheet 是 4 列二级导航，但移动端底部已有 4 个一级导航。为什么导航分两处？如果 8 项全放底部可滚动，或 More sheet 改全屏 nav 覆盖层，会不会更直接？
